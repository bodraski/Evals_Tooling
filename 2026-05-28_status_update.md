# Reasoning в vLLM + Inspect AI

## Как работает парсинг reasoning

При получении ответа от модели есть два слоя обработки:

**Слой 1 — vLLM сервер** (контролируется флагом `--reasoning-parser`):
- без флага → `<think>` теги остаются в `content` как plain text, поле `reasoning: null`
- с флагом → reasoning в отдельном поле `reasoning`

**Слой 2 — Inspect AI** (fallback парсер):
- читает `content`, ищет `<think>` теги, вырезает их
- итоговый `completion` для scorer получается норм в обоих случаях

## Кейсы по конфигурациям

| Конфигурация | `<think>` в content | `reasoning` поле | Inspect AI completion | Проблема |
|---|---|---|---|---|
| Внешний провайдер | нет | зависит от провайдера | чистый | нет |
| vLLM + `--reasoning-parser` | нет | заполнен | чистый | нет — но Inspect не читает `reasoning` поле |
| vLLM без флага | всегда | null | чистый | нет — Inspect парсит сам через fallback |
| Inspect-managed vLLM | зависит от PR | зависит от PR | — | решается PR |


## Silent failure modes

**Реальный:** fallback парсер срабатывает на любую модель - если обычная модель возвращает текст с `<think>` (например, отвечает на вопрос про этот тег), Inspect вырезает часть ответа в reasoning блок. Ошибки нет, ответ неполный.

**Решение в предложенном PR:** ограничить fallback известными семействами по имени модели.

## Открытые вопросы

**1. Вопрос в PR с маппингом по имени модели**

Qwen3 (а может и у других моделей) существует в двух режимах — с thinking и без (`enable_thinking=False`). Имя модели одинаковое. Regex `Qwen3` срабатывает в обоих случаях. Хуже: если промпт содержит `<think>` и модель его воспроизводит в ответе — fallback вырежет кусок ответа.

Вопрос: достаточно ли надёжен маппинг по имени как guard?

**2. Нужен ли fallback вообще?**

Есть вопрос об отказе от fallback целиком в пользу warning. Как надёжно детектировать что сервер настроен некорректно? Один из вариантов - через `system_fingerprint` (vLLM всегда его отдаёт, внешние провайдеры - нет).

**3. Вопрос по скоупу**

Неясно на какой кейс ориентируются мейнтейнеры в первую очередь:

- Inspect-managed vLLM (inspect eval --model vllm/...): Inspect сам запускает сервер, может автоматически проставить правильные флаги. Этот кейс закрывается PR.
- User-managed vLLM (пользователь поднял сервер сам, Inspect подключается по API): для Inspect это просто OpenAI-compatible endpoint, бэкенд неизвестен. Ответственность за конфигурацию на пользователе.

Второй кейс реален для облачного сэтапа (GPU сервер отдельно, Inspect на локальной машине без GPU) — что вероятно типично для исследователей. Но мейнтейнеры явно не обозначили, надо ли его рассматривать.

Вопрос: нужна ли документация для user-managed сценария, или фокус только на Inspect-managed?
