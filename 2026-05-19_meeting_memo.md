## Suggested Project Ideas

###  🔄 Импорт траекторий кодинг-агентов
- Пайплайн по импорту траекторий из Cursor, Claude Code, Codeux и прочих
- **Важно:** чтобы всё чётко работало и было ясно, где subagent, где compaction и т.п.

**​Плюсы:**
- High-value проект.
- Норм параллелится:
    - ведущие инженеры проектируют общую архитектуру ядра парсера и схему данных
    -  инженеры реализации пишут конкретные адаптеры под каждый инструмент.
      
**​Минусы:**
- слабая стандартизация логов закрытых инструментов
- рутина с JSON

​Стек: Python (Pydantic для валидации схем, Click/Typer для CLI), JSON/YAML-парсеры.

### Доступы и Риски (РФ):
- Низкие. Доступ к API моделей не нужен — работа с готовыми файлами логов.
- Единственная сложность — выкачать сами файлы логов/траекторий из-за платных или заблокированных аккаунтов Cursor/Claude Code.
  
### Ресурсы
- Оценка сложности: 6/10
- ​Итоговая оценка: 8.5 / 10
- ​Оценка времени: ~160–200 часов.
- ​Распределение:
      - ~40 ч. (Ведущие) + ~140 ч. (Прикладные)

### ⚡ Real-time вывод моделей
- Функционал, поторый не помешал бы в Apollo (и не только!):
  - из Inspect получать вывод моделей **в реальном времени**, а не когда всё готово и эвал завершен
    
**Уточнение постановки задачи**
- Отображать выполненные задачи в интерфейсе сразу после того, как по ним отработал скорер.
- ⚡ Пошаговое обновление интерфейса (Event-Driven UI, не live-стриминг)


**​Плюсы:**
​- UX-буст: видно результаты первых задач уже через 15 секунд. Если промпт «поплыл», эвал можно отменить сразу, сэкономив деньги и время.
​- Хороший баланс для мкиллов команды: 
   - ведущие инженеры делают асинхронный эмиттер событий на Python
   - инженеры реализации получают большой и понятный фронтенд-кусок по динамической отрисовке данных
- Безопасность архитектуры: не трогаем генерацию токенов моделей, а перехватываем уже готовый, посчитанный скорером объект задачи.

​**Минусы:**
- ​Необходимость глубже разобраться в том, как Inspect сейчас буферизирует и пишет файлы логов (.inspectlog), чтобы вклиниться в этот процесс.

  
​Стек: Python (asyncio, WebSockets или Server-Sent Events для отправки готовой задачи), TypeScript, React (для реактивного обновления дашборда «на лету»)


### Доступы и Риски (РФ)
- Низкие
- Все тесты отправки событий на фронтенд можно проводить локально на mock-данных, не тратя баланс API.
