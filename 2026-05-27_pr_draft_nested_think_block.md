# Черновик PR: парсинг вложенных блоков `<think>`

# https://github.com/UKGovernmentBEIS/inspect_ai/compare/main...gornkv:inspect_ai:nested_think_block

## Контекст

В Inspect уже есть fallback-парсер для reasoning, который модель возвращает прямо в тексте assistant-сообщения в виде `<think>...</think>`. Он нужен для OpenAI-compatible серверов и локальных inference-стеков в случаях, когда reasoning приходит не в структурированном API-поле вроде `reasoning_content` или `reasoning_details`, а внутри обычного поля `content`.

Существующий парсер использовал одно нежадное регулярное выражение:

```python
r"<think([^>]*)>(.*?)</think>"
```

Оно работает для простого случая:

```text
<think>reasoning</think>answer
```

но ломается, если внутри reasoning-текста встречается еще один блок `<think>...</think>`.

## Механизм проблемы

Ошибка возникает из-за сочетания нежадного regex и вложенных тегов.

Для такого входа:

```text
<think>Outer <think>inner</think> continues</think>Visible text
```

предыдущий regex начинал матч с первого открывающего `<think>` и останавливался на первом закрывающем `</think>`. Но этот первый закрывающий тег относится ко внутреннему блоку, а не к внешнему.

В результате парсер извлекал:

```text
Outer <think>inner
```

и оставлял в видимом ответе:

```text
 continues</think>Visible text
```

Это создает две пользовательски видимые проблемы:

1. Reasoning capsule оказывается обрезанной.
2. Часть приватного reasoning и лишний закрывающий тег могут утечь в обычный assistant content.

Такое может случиться даже если модель не пыталась намеренно сгенерировать некорректную XML-подобную разметку. Модели, обученные reasoning через теги, могут цитировать, повторять или описывать сам формат тегов внутри скрытого reasoning-текста. Чистый regex-парсер не может отличить закрывающий тег внутреннего блока от закрывающего тега внешнего блока.

## Изменение

PR добавляет небольшой scanner перед существующим regex fallback.

Новый helper последовательно проходит по токенам `<think...>` и `</think>`, считает глубину вложенности и возвращает первый сбалансированный внешний блок:

```python
def _find_nested_think_block(content: str) -> re.Match[str] | None:
    start: int | None = None
    depth = 0
    pattern = re.compile(r"<think([^>]*)>(.*)</think>", re.DOTALL)

    for match in re.finditer(r"<think([^>]*)>|</think>", content, re.DOTALL):
        tag = match.group(0)
        if tag.startswith("<think"):
            if start is None:
                start = match.start()
                depth = 1
            else:
                depth += 1
        elif start is not None:
            depth -= 1
            if depth == 0:
                return pattern.search(content, start, match.end())

    return None
```

`parse_content_with_reasoning()` теперь сначала пробует этот scanner:

```python
match = _find_nested_think_block(content) or re.search(pattern, content, re.DOTALL)
```

Остальной путь парсинга не меняется: атрибуты вроде `signature`, `redacted` и `format` по-прежнему обрабатываются существующим кодом, а функция по-прежнему возвращает очищенный видимый content и `ReasoningCapsule`.

## Почему выбрано такое решение

Решение намеренно узкое. Баг именно в поиске закрывающего тега, соответствующего первому блоку `<think>`. Счетчик глубины решает эту задачу без изменения публичной модели данных, без изменения поведения провайдеров и без добавления общего HTML/XML-парсера.

Также сохраняется старое поведение для malformed content. Если scanner не находит сбалансированный внешний блок, парсер откатывается на предыдущий regex. Это сохраняет существующий best-effort подход для случаев вроде несбалансированного вложенного блока, где простое извлечение до первого закрывающего тега все еще лучше, чем отсутствие результата.

Реализация остается локальной в `_reasoning.py`, где и находится исходный parser. Ответственность за парсинг не переносится в код OpenAI-compatible провайдеров, поэтому все call sites, которые уже используют `parse_content_with_reasoning()`, получают исправление одинаково.

## Рассмотренные альтернативы

### Оставить исходный regex

Это минимальное изменение, но оно оставляет баг с утечкой и обрезкой reasoning. Regex без рекурсии или ручного прохода по глубине не может корректно сопоставлять вложенные пары тегов.

### Использовать жадный regex

Жадный pattern матчился бы до последнего `</think>` вместо первого. Это исправляет простой nested example, но ломается, если в answer text или последующем content встречается еще один `</think>`. Также такой подход усложняет поведение для malformed content: один лишний закрывающий тег может захватить слишком большой фрагмент текста.

### Использовать XML/HTML parser

Content не гарантированно является валидным XML или HTML. Reasoning text может содержать произвольный model output, частичные теги, escaped snippets, Markdown, код или malformed tag-like text. Строгий parser либо отклонял бы полезные случаи, либо требовал бы дополнительной recovery-логики. Browser-style HTML parser тоже мог бы нормализовать или переинтерпретировать текст способом, который не подходит для сохранения model output.

### Парсить все reasoning blocks

Текущий контракт извлекает одну reasoning capsule из одной строки assistant message content. Расширение этого поведения до нескольких capsules было бы более крупным behavioral change с последствиями для display, logging и downstream consumers. PR сохраняет существующий контракт и исправляет только поиск границы первого блока.

## Тесты

PR добавляет тесты на:

- сбалансированный вложенный блок;
- вложенный блок с лишним закрывающим тегом после него;
- malformed nested content, где по-прежнему используется fallback-поведение;
- unclosed nested content, где сохраняется старое best-effort поведение regex.

Эти тесты находятся в `tests/model/test_reasoning_parse.py`.

## Риск

Главный риск в том, что входы с необычным tag-like text внутри reasoning теперь могут быть распознаны как один более крупный сбалансированный блок вместо прежнего обрезанного блока. Это и есть ожидаемое исправление. Поведение для существующего malformed input сохраняется через regex fallback, поэтому изменение должно быть низкорисковым для content без вложенных тегов.
